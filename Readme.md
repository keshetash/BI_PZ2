## Цель работы

Научиться создавать системы работы с потоками данных с использованием инструмента Apache Airflow.

---

## Задачи

1. Развернуть приложение Apache Airflow в среде Linux (Ubuntu)
2. Разработать три DAG (Directed Acyclic Graph) для автоматизации бизнес-процессов сервиса по ремонту техники
3. Поставить DAG'и на выполнение в Apache Airflow и настроить расписание
4. Протестировать работу пайплайнов и проанализировать результаты


## Описание предметной области

### Сервис по ремонту техники

Система управления ремонтными заказами, которая обрабатывает данные о:

#### Сущности предметной области:

1. **Клиенты**
   - ФИО
   - Телефон
   - Email
   - Адрес
   - Дата регистрации

2. **Виды устройств**
   - Смартфоны
   - Ноутбуки
   - Планшеты
   - ПК
   - Телевизоры

3. **Типы неисправностей**
   - Аппаратные (разбит экран, не включается)
   - Программные (сбой ПО, вирусы)
   - Механические (повреждение корпуса)

4. **Инженеры**
   - ФИО
   - Специализация
   - Статистика работы

5. **Заказы на ремонт**
   - Статус (принят, в работе, готов, выдан)
   - Сроки выполнения
   - Стоимость
   - Приоритет

6. **Запчасти**
   - Наименование
   - Цена
   - Количество на складе
   - Поставщик

### Этап 2: Разработка DAG'ов

#### DAG #1: Обработка заказов на ремонт (repair_orders_etl)

**Назначение:** ETL-пайплайн для обработки новых заказов на ремонт техники

**Расписание:** Каждые 4 часа

**Задачи:**

1. **extract_orders** — извлечение данных о новых заказах
   - Подключение к базе данных сервиса
   - Извлечение заказов за последние 24 часа
   - Вывод информации о заказах

2. **transform_orders** — обработка и категоризация заказов
   - Анализ типов неисправностей
   - Определение сложности ремонта
   - Расчет предварительной стоимости

3. **load_to_system** — загрузка обработанных данных
   - Обновление статусов заказов
   - Назначение инженеров
   - Резервирование запчастей
   - Отправка уведомлений клиентам

**Структура пайплайна:**
```
extract_orders → transform_orders → load_to_system
```

#### DAG #2: Аналитика и отчетность (repair_analytics_pipeline)

**Назначение:** Генерация и анализ данных для отчетности сервиса

**Расписание:** Ежедневно (daily)

**Задачи:**

1. **generate_clients** — генерация данных о клиентах (20 записей)
2. **generate_devices** — генерация данных об устройствах (30 записей)
3. **generate_orders** — генерация заказов на ремонт (50 записей)
4. **generate_parts** — генерация данных о запчастях (40 записей)
5. **analyze_data** — комплексный анализ данных сервиса
6. **generate_report** — формирование итогового отчета

**Структура пайплайна:**
```
[generate_clients, generate_devices] → generate_orders → [generate_parts, analyze_data] → generate_report
```

**Особенности:**
- Использование **XCom** для передачи данных между задачами
- **Параллельное выполнение** задач на разных этапах
- Использование библиотеки **pandas** для обработки данных
- Использование библиотеки **faker** для генерации реалистичных данных

**Пример аналитики:**

#### DAG #3: Мониторинг работы инженеров (engineer_performance_monitor)

**Назначение:** Мониторинг производительности инженеров и загруженности сервиса

**Расписание:** 3 раза в день (9:00, 14:00, 18:00)

**Задачи:**

1. **monitor_engineers** — мониторинг статистики работы инженеров
   - Количество завершенных заказов
   - Количество заказов в работе
   - Среднее время ремонта

2. **check_workload** — проверка загруженности сервиса
   - Общее количество заказов
   - Активные заказы
   - Заказы в ожидании запчастей
   - Расчет процента загруженности

3. **send_notifications** — отправка уведомлений
   - SMS клиентам
   - Email уведомления
   - Push-уведомления инженерам

**Структура пайплайна:**
```
[monitor_engineers, check_workload] → send_notifications
```

**Особенность:** Параллельное выполнение мониторинга и проверки загруженности для ускорения работы пайплайна.
